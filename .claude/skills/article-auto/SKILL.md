# 記事1本フルオート作成ワークフロー

**案出し(`article-ideas`)→ 1件を自動選定 → 作成・公開(`article-workflow` オートモード)**
を、ユーザーへの確認を挟まず一気通貫で実行するオーケストレーションスキル。

このスキル自体は新しい執筆ルールを持たない。案出しと作成の実体はそれぞれ
[[article-ideas]] と [[article-workflow]] スキルに委ね、本スキルは
**「案を1件自動で決めて、そのまま最後まで自動で走らせる」** 連結だけを担う。

> **言語ルール:** ユーザーへの発信(選定理由・進捗報告・PR 説明文・コミットメッセージなど)は
> すべて **日本語** で行う(`CLAUDE.md`)。

## 前提として必ず読むもの

- `docs/rules/article/common.md` — 技術記事作成 共通ガイドライン(リポジトリ前提・テーマの焦点・
  量産型回避・既存記事の把握方法)。案の選定と重複回避はこれに準拠する。

執筆・レビュー・PR の詳細ルールは `article-workflow` が読み込むため、本スキルで重ねて読む必要はない。

---

## フルオートの原則

- **ユーザーへの確認(`AskUserQuestion`)を挟まない。** 実行モードの確認・案の選定確認・
  PR 前の内容確認は、いずれも行わずに進める。
- ただし `article-workflow` に定める **不可逆操作の安全弁は維持する**。
  すなわち、修正方針に重大な曖昧さがある場合や破壊的・不可逆な判断が必要な場合、
  および `article-workflow` の「マージ直前チェックリスト」でサーバー側ガードが無く自動マージが
  安全でないと判断される場合は、フルオートであってもユーザーへ確認・報告して判断を仰ぐ。
- テーマが定まらない(案が1件も作れない、または選定基準を満たす案が無い)場合は、
  **案の再生成を試みる。案出し(Step 1)は最大 3 回まで許容する。**
  3 回試みても選定基準を満たす案が得られないときにのみ、停止してユーザーへ確認する。

---

## 手順

### Step 1. 案を出す(`article-ideas`)

> **リトライ制御:** Step 1 は「試行回」として数える。**最大 3 回**まで実行してよい。
> 各回のはじめに現在の試行回数(1〜3)を把握しておく。

1. `article-ideas` スキルの手順に従い、既存記事(`_articles_en/` / `_articles_ja/` の slug)を
   実ファイルから把握したうえで、重複しない記事案を **5〜10 件** 生成する。
   - フルオートでは大量の案を提示する必要はない。**選定に足る少数(既定 5 件)** で足りる。
   - 各案は `article-ideas` の出力形式(テーマ / slug / category / 実行モード)を満たす完結した指示文にする。
   - ユーザーへ案の一覧を提示して選ばせることはしない(確認を挟まないため)。
   - **再生成する回では、これまでの全試行で挙げた案を累積して除外し、別の切り口・テーマで生成する。**
     直前の試行分だけでなく、過去に一度でも挙げた案(テーマ・slug)をすべて記録しておき、
     3 回目でも 1・2 回目の案が再登場しないようにする(同じ案を繰り返さない)。

### Step 2. 1件を自動選定する

1. Step 1 の案から **1件を自動で選ぶ。** 選定基準は次の優先順で判断する。

   1. `common.md` のテーマ焦点(.NET / C# / WPF、トラブル解決・実装解説型、1記事1テーマ)への適合度。
   2. 実務での遭遇頻度・検索需要が高く、独自の背景・落とし穴・比較を書きやすい(AdSense 適合性を満たしやすい)テーマ。
   3. 既存記事と slug・内容が重複せず、独自性を確保しやすいもの。

   同程度の候補が並ぶ場合は、上記により合致するものを1件選ぶ(恣意的な固定ではなく、基準に基づいて決める)。

   - **選定基準を満たす案が1件も無い場合(テーマが定まらない場合)は Step 1 へ戻って再生成する。**
     ただし Step 1 の試行は **最大 3 回まで**。
     - 試行が 3 回未満なら Step 1 へ戻り、別テーマで案を生成し直す。
     - **3 回目でも選定基準を満たす案が得られないときにのみ**、以降を中止して
       状況(3 回試みたが適切なテーマが無かった旨)を日本語で報告し、ユーザーの判断を仰ぐ。
   - 選定基準を満たす案が得られたら Step 3 へ進む。

2. **選定結果を日本語で1〜2行報告する。** 何を(テーマ・slug・category)なぜ選んだか(選定理由)を
   簡潔に記す。報告はするが、承認は求めずそのまま Step 3 へ進む。

### Step 3. 記事を作成・公開する(`article-workflow` オートモード)

1. `article-workflow` スキルを **オートモード(`RUN_MODE = auto`)** で起動する。
   - Step 2 で選んだ案を **テーマとして渡す**(`article-workflow` Phase 0 の「テーマ確認」はこれで満たされる)。
   - **`RUN_MODE = auto` を確定値として引き渡す(引き渡し契約)。** `article-workflow` Phase 0 手順 1 は
     実行モードの確認を必須とするが、これは「モードが未確定なら確認する」という要求であって、
     モードが確定していれば確認そのものを省略できる。本スキルは起動時点でフルオートが確定しているため、
     `RUN_MODE = auto` を **既に確定した選択肢として明示的に注入**する。すなわち `AskUserQuestion` は
     行わないが、`RUN_MODE` を未設定のままにはしない — `auto` を確定値として与えたうえで Phase 1 から進める。
     この注入により Phase 0 の必須確認の趣旨(モードの確定)は満たされ、下流で `RUN_MODE` が
     未定義になることはない。
   - 以降は `article-workflow` の定義どおり:
     - Phase 1: 日英ペアの記事を作成(ルール準拠)。
     - Phase 2: サブエージェントによる文体・構成レビュー + 技術検証専任パスを **指摘ゼロまで** 反復。
     - Phase 3: `auto` のため **内容確認をスキップ** して Phase 4 へ。
     - Phase 4: PR 作成 → CI 監視 → CodeRabbit 再レビュー → 指摘・失敗ゼロまで修正ループ、
       対応済みスレッドを解決済みに。
     - Phase 5: マージ直前チェックリストを満たしてからマージ → head ブランチは自動削除 → 購読解除。

2. **完了報告。** `article-workflow` Phase 5 に従い、記事タイトル・PR 番号・マージ済みである旨を
   日本語で報告する。あわせて、本スキルが選定した案(Step 2 の選定理由)も併記する。

## 全体フロー要約

```text
Step 1  article-ideas で案を 5〜10 件生成(既存 slug と重複回避)  ┐ 選定基準を満たす案が
Step 2  基準に基づき 1 件を自動選定 → 選定理由を日本語で報告      ┘ 無ければ Step 1 へ戻る
          (承認は求めない / Step 1 の再生成は最大 3 回、3 回目でも不可なら停止して確認)
Step 3  article-workflow を RUN_MODE=auto で起動(モード確認はスキップ、選定案をテーマに)
          Phase 1 作成 → Phase 2 反復レビュー → Phase 3(auto=確認スキップ)
          → Phase 4 PR・CI・再レビュー → Phase 5 マージ・完了報告
```

## 判断の原則

- **確認は挟まない**が、`article-workflow` の不可逆操作の安全弁(曖昧・破壊的な判断、
  マージ安全性)は維持する。ここに該当したときだけ停止してユーザーに確認する。
- 案の選定は**基準に基づいて**行い、選定理由を必ず報告する(ブラックボックスにしない)。
- 記事の品質要件(独自性・最小文量・AdSense 適合性)は `article-workflow` に委ねる。
  フルオートでも品質基準を下げない。
- **案が1件も作れない場合、または選定基準を満たす案が無い場合(いずれもテーマが定まらない場合)は、
  案出し(Step 1)を最大 3 回まで再試行する。** 3 回試みても選定基準を満たす案が得られないときにのみ、
  記事作成へ進まず停止して確認する(この停止条件はフルオートの原則・Step 2 の定義と一致させる)。
