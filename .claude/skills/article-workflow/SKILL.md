# 記事作成・公開ワークフロー

技術記事の **作成 → レビュー → 確認 → PR → CI 監視 → マージ → ブランチ削除** を一気通貫で実行するスキル。

> **言語ルール:** ユーザーへの発信(質問・進捗報告・PR 説明文・コミットメッセージなど)はすべて **日本語** で行う(`CLAUDE.md`)。

## 前提として必ず読むルール

作業前に以下を読み、記事はこれらに完全準拠させること。

- `docs/rules/article/common.md` — 技術記事作成 共通ガイドライン(リポジトリ前提・格納先・テーマの焦点・量産型回避・既存記事の把握方法など、記事関連スキル共通の土台)
- `docs/rules/article/guidelines.md` — 執筆方針・文体・必須要素
- `docs/rules/article/template-ja.md` / `template-en.md` — 記事テンプレート
- `docs/rules/article/review-checklist.md` — 公開前チェックリスト
- `.markdownlint.json` — 有効な lint ルール(`MD060` は無効)

記事の格納先とフロントマター:

- 日本語記事: `_articles_ja/<slug>.md`(`layout: article-ja`)
- 英語記事: `_articles_en/<slug>.md`(`layout: article-en`)
- 既存記事はすべて日英で対になっている。**原則、日本語版と英語版を両方作成する**。
- フロントマターは `title` / `date`(UTC) / `category` / `excerpt`(160字以内) / `layout` を設定する。
- 日英の対応記事間には相互リンクを **張らない**(`guidelines.md` §7)。

## Phase 0. 起動時: モードとテーマの確認

1. **実行モードの確認(必須)** — ただし **モードが既に与えられている場合は確認を省略する**。

   起動時の **スキル引数や直前の会話でモードが明示されている場合**(例: `article-auto` などの
   上位スキルから `RUN_MODE=auto` が渡された、ユーザーが「オートで」と指定した)は、その値を
   `RUN_MODE` に採用し、`AskUserQuestion` は行わない。引数として受け取る値は
   `RUN_MODE=auto` / `RUN_MODE=manual` / `RUN_MODE=pre_pr` の形式で、下記の 3 モードに対応する。
   **モードが与えられていない場合にのみ**、`AskUserQuestion` で以下の 3 択を問う。

   > 質問例: 「このワークフローをどのモードで実行しますか?」
   >
   > - **オートモード** — PR 作成前の内容確認をスキップし、レビューが通り次第そのまま PR・CI・マージまで自動で進める。
   > - **手動モード(既定)** — PR 作成前に記事内容の確認を依頼する。承認後に PR・CI・マージへ進める。
   > - **PR 前まで** — 記事の作成・反復レビュー(Phase 2)・ブランチ作成を行い、コミット＆push まで
   >   実施して停止する。PR 作成以降(Phase 4・5)は **別途指示があるまで一切行わない**。

   採用した値(引数由来・確認由来のいずれも)を `RUN_MODE = auto | manual | pre_pr` として以降の判断に使う。
   - `auto` … Phase 3 の確認をスキップし、Phase 4・5 まで自動で進める。
   - `manual`(既定) … Phase 3 で内容確認を依頼し、承認後に Phase 4・5 へ進む。
   - `pre_pr` … Phase 1(記事作成・ブランチ作成)と Phase 2(反復レビュー)を実施し、
     Phase 3 で変更をコミット＆push して停止する。
     **Phase 4・5(PR 作成・CI 監視・マージ)は行わず、ユーザーからの明示的な指示を待つ**。

2. **記事テーマの確認** — スキル引数や直前の会話でテーマが与えられていればそれを使う。
   不明な場合のみ、テーマ(扱う技術・解決する問題)を短く確認する。`RUN_MODE == auto`(オートモード)でも、テーマが無ければここは確認する(テーマが無ければ記事は書けないため)。

## Phase 1. 記事の作成

1. **ブランチを用意する。まず必ず最新の `main` を取得してから開始する。**
   `git fetch origin` で最新のリモート追跡ブランチを取得する。
   そのうえで、**現在どのブランチにいるかではなく、作業ブランチの存在有無で分岐する**:
   - **存在しない場合(新規作成)**: `git switch -c <branch> origin/main` で最新 `main` から作成する。
   - **既に存在する場合**: `git switch <branch>` で切り替えてから、**ブランチを他者と共有しているかで更新方法を選ぶ**(判断基準は push 済みか否かではなく、共有の有無):
     - **自分専用のブランチ(他者と共有していない。push 済みでも可)**: `git rebase origin/main` で最新 `main` に追随する。
       rebase 後、既に push 済みの内容を更新するには `git push --force-with-lease` を用いる(通常の push は拒否される)。
     - **他者と共有中のブランチ**: 履歴を書き換えない `git merge origin/main` を用いる(rebase および force push は行わない)。

     いずれの場合も `git checkout -B <branch> origin/main` は使わない(既存ブランチの未 push コミットが参照から外れるため)。
   これにより直近でマージされた記事・設定・スキルを取り込んだ状態から作業でき、重複提案や古い前提での作業を避けられる。
   ブランチ名はテーマが分かる `slug` を用いる(例: `claude/article-<slug>`)。
   > いずれの場合も、取得せず古い状態のまま進めない。競合が発生した場合は解消してから進める。

2. **記事タイプを選ぶ。**「問題解決型(§2.1)」か「手順・解説型(§2.2)」かを決める。

3. **記事を執筆する。**`template-ja.md` / `template-en.md` の構成に沿って日英両方を書く。
   - 文体は断定的・無主語構文(常体)。会話的表現・Q&A 見出しを避ける。
   - すべてのコードブロックの前後に説明文を付ける。
   - 概要・まとめは省略しない。コードを伴う場合は実装例も省略しない。
   - AdSense 適合性(独自性・最小文量・比較表/落とし穴/選択指針のいずれか)を満たす。
   - 文末ごとに改行を入れる(読点・カンマでは改行しない)。

4. **lint を確認する。** markdownlint が利用可能なら実行し、警告が無いことを確認する
   (`npx -y markdownlint-cli2 "<file>"` など)。利用不可の場合はチェックリストの目視確認で代替する。

## Phase 2. サブエージェントによる反復レビュー(指摘ゼロまで)

記事の下書きに対し、**指摘が無くなるまで** 以下を繰り返す。
本リポジトリでは Copilot の自動レビューを使わない方針のため、**Copilot が担っていた構文・正確性の検証はこの Phase 2 と
Phase 4 の CodeRabbit で補完する**。特に技術検証(手順 2)は、公開後のボット任せにせず PR 前にここで確実に潰す。

1. **文体・構成レビュー用サブエージェントを起動する。**`Agent` ツール(`subagent_type: "general-purpose"` など)で、
   対象記事ファイルと以下のルールを渡してレビューさせる。
   - `docs/rules/article/review-checklist.md` の全項目
   - `docs/rules/article/guidelines.md` の文体・構成・必須要素
   - フロントマターの妥当性、markdownlint 準拠

   > サブエージェントへの指示は「**指摘は件数を絞らず、見つけたものをすべて挙げること**」を明記する
   > (`.github/copilot-instructions.md` のコードレビュー方針に従う)。日英それぞれをレビューさせる。

2. **技術検証専任のサブエージェントを別途起動する(文体レビューとは別枠の独立した検証パス)。**
   文体・構成の観点で正確性の検証が薄まらないよう、技術的な正しさだけを見る検証パスを独立して回す。
   検証対象は記事本文に限らず、記事が含むすべてのコード例・技術的主張とする。
   - **すべてのコード例(XAML・C#・シェル等)と記載した規則を、公式ドキュメントで裏取りすること。**
     .NET / WPF 等は `Microsoft_Learn` MCP(`microsoft_docs_search` / `microsoft_docs_fetch`)を用いて一次情報で確認する。
   - **出典を挙げずに「問題なし」としない。** 「確認済み」と述べるには、根拠となる公式ドキュメントの該当箇所を示すこと。
     (過去に、裏取りせず誤った主張を『確認済み』としてしまい、公開後のレビューで指摘された実例がある。これを防ぐための規則。)
   - 特に次を必須チェックする:
     - **構文・エスケープ・パーサー規則**(例: XAML の `{Binding ...}` ショートハンドでの先頭 `{` の `{}` エスケープ、
       カンマ・空白などの区切り文字の引用符保護。要否の条件を取り違えていないか)。
     - **API の挙動・既定値**、および**機能が導入されたバージョン**の事実主張。
     - コード例が実際にコンパイル/パース/実行できるか。
   - これらの検索・裏取り(`Microsoft_Learn` 等)は**確認・報告を挟まず実行する**。
     途中経過(検索している旨・処理中である旨)をユーザーへ逐一報告しない。
   - 指摘は件数を絞らず、見つけたものをすべて挙げる。

3. **指摘を修正する。** 文体レビュー・技術検証の両方から返ってきた指摘をすべて記事へ反映する。

4. **再レビューする。** 修正後の記事を再び両サブエージェントへ渡す。
   **新たな指摘が無くなるまで** 1〜4 を繰り返す。

5. レビューが収束したら、次フェーズへ進む前に主要な変更点を簡潔にまとめておく。

## Phase 3. モード別分岐(内容確認 / PR 前停止)

`RUN_MODE` によって分岐する。

- `RUN_MODE == manual`(既定)の場合:
  - 記事の要点(タイトル・構成・主な主張・日英差分)を提示し、`AskUserQuestion` で確認を依頼する。
    - **承認** — このまま Phase 4(PR)に進む。
    - **修正あり** — 指摘を受けて Phase 2 のレビュー修正へ戻り、再度確認を依頼する。
  - 承認が得られるまで PR は作成しない。

- `RUN_MODE == auto` の場合:
  - この確認は **スキップ** し、そのまま Phase 4 へ進む。

- `RUN_MODE == pre_pr` の場合:
  - **PR は作成しない。** 記事の変更をコミットし、`git push -u origin <branch>` で作業ブランチへ push して停止する。
    push は Web 実行環境で作業を失わないための保全であり、PR ではない。
    ネットワークエラー時は 2s/4s/8s/16s の指数バックオフで最大 4 回リトライする。
  - コミット完了後、記事の要点・ブランチ名・コミット概要をユーザーへ日本語で報告し、
    「PR 以降は指示待ち」である旨を明示する。
  - **Phase 4・5(PR 作成・CI 監視・マージ)は、ユーザーからの明示的な指示があるまで一切行わない。**
    指示があった時点で Phase 4 から再開する。

## Phase 4. PR 作成と CI 監視(指摘ゼロまで)

1. **コミット & プッシュ。** 変更をコミットし、`git push -u origin <branch>` する
   (ネットワークエラー時は 2s/4s/8s/16s の指数バックオフで最大 4 回リトライ)。
   - `pre_pr` からの再開などで **追加の変更が無い場合はコミットを省略する**(`nothing to commit` で止めない)。
     ただし未 push のコミットが残っていれば `git push -u origin <branch>` を実行してから手順 2(PR 作成)へ進む。
     追加修正がある場合はコミットしてから push する。

2. **PR を作成する。** GitHub MCP(`mcp__github__create_pull_request`)で `main` 宛の PR を作成する。
   - リポジトリに PR テンプレート(`.github/pull_request_template.md` 等)があれば、その見出し構成に沿って本文を埋める。
   - タイトル・本文は日本語で記述する。

3. **CI を監視する。**`subscribe_pr_activity` で PR を購読し、以降のイベント
   (CI 結果・レビューコメント)を受け取る。

4. **指摘・失敗を解消する。** CI 失敗またはレビュー指摘があるたびに:
   - 原因を診断して記事・設定を修正する。
   - コミット & プッシュし、CI を再実行(必要なら `mcp__github__actions_run_trigger` や再 push で再起動)する。
   - **CI がすべてグリーン、かつ未解決の指摘が無くなるまで繰り返す。**
   - 修正の解釈に**重大な**曖昧さがある場合や大規模改修が必要な場合は、`AskUserQuestion` で確認する
     (オートモードで確認を残すのはこの場合に限る。マージ操作そのものは Phase 4 手順 8 の基準で
     自動判断し、内容に重大な曖昧さが無ければ確認しない)。

5. **対応を終えたレビュースレッドは解決済みにする。** 各レビュー指摘は、対応が終わった時点で
   `mcp__github__resolve_review_thread` を使ってスレッドを **解決済み(resolved)** にする。
   - **修正で対応した指摘** — 修正をコミット・プッシュしたうえで、該当スレッドを解決済みにする。
   - **反論・見送りで対応を終えた指摘**(誤検知、方針上採用しない等) — その理由を
     `mcp__github__add_reply_to_pull_request_comment` で **1 回だけ** 簡潔に返信し、
     合意が取れた/対応方針が確定した時点でスレッドを解決済みにする。
   - スレッド ID は `mcp__github__pull_request_read`(`method: "get_review_comments"`)で取得する
     (各スレッドの `id`)。修正で行が変わると `is_outdated` にはなるが、`is_resolved` は
     手動で解決するまで `false` のままである点に注意する。
   - **マージ前に、未解決(`is_resolved: false`)のレビュースレッドが残っていないことを確認する。**

   > 本リポジトリの PR には `Lint Markdown`(`.github/workflows/lint-markdown.yml`)が走り、
   > markdownlint(`.markdownlint.json`)でリポジトリ全体の Markdown を検査する。ローカルでも
   > `npx -y markdownlint-cli2 "**/*.md"` で同じ検査を再現できる。
   > CI がグリーンになるまで修正・再 push を繰り返す。

6. **修正 push のあとは CodeRabbit へ再レビューを要求する。** `Lint Markdown`(CI)は
   `pull_request` トリガーで、Markdown 関連ファイル(`**/*.md` / `.markdownlint.json` / 当該ワークフロー)を
   変更した push でのみ自動再実行される(対象外の push では check run が作成されないため、未起動を許容する)。
   一方、**CodeRabbit は `.coderabbit.yaml` で `auto_incremental_review: false` としているため、
   push だけでは自動再レビューされない**。
   そのため、レビュー指摘を修正して push したら、PR に `@coderabbitai review` とコメントして
   明示的に再レビューを要求する。
   - **本リポジトリでは Copilot の自動レビューを使わない方針である。** Copilot への再レビュー要求
     (`mcp__github__request_copilot_review`)は行わない。Copilot が担っていた構文・正確性の検証は、
     PR 前の Phase 2 技術検証パスと、CodeRabbit(`profile: assertive` + `path_instructions`)に集約している。
   - 再レビューで新たな指摘が来た場合は、手順 4〜5 と同様に修正・返信し、対応済みスレッドを
     解決済みにする。
   - **新たな指摘が無くなるまで**、「修正 push → 再レビュー要求 → 対応・解決」を繰り返す
     (直近コミットに対する未解決スレッドがゼロになるまで)。

7. **自動レビューボット(CodeRabbit 等)の完了を必ず待つ。** 本リポジトリでは
   `coderabbitai` などの自動レビューボットが動作する。これらは再レビュー要求後に処理が走り、
   完了まで数分かかる。**「review in progress」「Currently processing…」
   といった処理中表示が出ている間は、絶対にマージしない。**
   - CI がグリーンでも、`coderabbitai` の walkthrough や「Pre-merge checks passed」だけを
     根拠にマージしてはならない。これらは中間シグナルであり、その後に actionable な指摘が
     追加されることがある(実例あり)。
   - マージ可否は必ず、**そのボットの当該コミットに対するレビューパスが完了したこと**
     (処理中表示が消え、レビュー結果コメント/レビューが投稿済み)を確認してから判断する。
   - 判断が付かない場合は数分待って再確認する(セルフチェックインの再予約)。急いでマージしない。

8. **マージ直前チェックリスト(全項目を満たすまでマージ禁止)。**
   - [ ] 最新コミットに対する `Lint Markdown`(CI)が success。
         ただし `Lint Markdown`(`.github/workflows/lint-markdown.yml`)は
         `**/*.md` / `.markdownlint.json` / 当該ワークフローの変更時のみ起動する。
         **これらを変更しない PR ではチェックが起動しない**ため、その場合は check run 不在(未実行)を
         許容し、この項目は満たされたものとして扱う(記事 PR では必ず `.md` を変更するため通常は起動する)。
   - [ ] CodeRabbit 等の全自動レビューボットの再レビューが**完了**(処理中表示なし)し、
         actionable な未解決指摘がゼロ。
   - [ ] **全レビュースレッドが解決済みである**(作成者に関係なく `is_resolved == true`)。
   - [ ] **`CHANGES_REQUESTED` のレビューが残っていない。** `get_reviews` で、dismiss されていない
         レビュー提出に `state == CHANGES_REQUESTED` が 1 件も無いことを確認する。
         **本文だけで紐づくスレッドが無い `CHANGES_REQUESTED` レビューでもマージをブロックする**
         (スレッド解決の確認だけでは漏れるため)。この条件はマージ直前の再取得後にも再検証する。
   - [ ] **事前スナップショットを記録する。** マージ判定を始める前に、差分比較の基準として次を控える:
         現在の HEAD SHA、直近レビューコメントの ID、各 check run の ID と **status/conclusion**、
         各レビュースレッド(`get_review_comments`)の ID と **is_resolved**、
         各レビュー提出(`get_reviews`)の ID と **state**(`APPROVED` / `CHANGES_REQUESTED` / `DISMISSED` 等)。
         この基準が無いと「新しい変更」や「状態だけの変化」を判定できない。
   - [ ] **マージ実行の直前に最新状態を再取得し、スナップショットと照合する。**
         `mcp__github__pull_request_read`(`get_status` / `get_check_runs` / `get_review_comments` / `get_reviews`)
         で再取得し、各対象(check run・レビュースレッド・レビュー提出)の **完全な ID セットと状態** を突き合わせる。
         この確認から `merge_pull_request` 実行までの間に発生する TOCTOU(time-of-check-to-time-of-use)を
         避けるため、**新しい push・コミット・レビューコメント・check run の追加または削除、
         既存 check run の status/conclusion の変化、既存レビュースレッドの追加・削除・is_resolved の変化、
         既存レビュー提出の追加・削除・state の変化が 1 件でも見つかった場合は、
         マージを中止してチェックリストを最初からやり直す**。差分が無い場合のみ次へ進む。
   - [ ] **TOCTOU の権威はサーバー側の原子ガードに置く。** クライアント側の再取得＋事後検証
         だけでは、最終確認から `merge_pull_request` 実行までの間に push が入った場合に、未レビューの
         head がマージされるのを防げない(事後検出は `main` を保護しない)。現行の
         `mcp__github__merge_pull_request` は SHA 条件付けの引数を持たないため、真の原子性は
         **GitHub のブランチ保護に委ねる**: `main` に「必須ステータスチェック(CI と CodeRabbit の
         `Review completed` など)」と「Require branches to be up to date before merging」を設定する。
         これがあれば、確認後に head が変わったりチェックが未グリーンになった時点で
         `merge_pull_request` はサーバー側で拒否され、原子的に安全となる。
   - [ ] **`main` が保護ブランチ(`protected: true`)なら、オートモードでは確認を求めずに自動マージする。**
         `mcp__github__list_branches` 等で `main` の `protected` を確認し、`true` であれば原子ガードが
         構成されているとみなして `merge_pull_request` を実行してよい(必須ステータスチェックや
         up-to-date の詳細構成まで MCP で確認できなくても、`protected: true` を確証として扱い、
         この一点をもって自動マージを妨げない)。
         **`protected: false`(保護なし)の場合に限り**、原子性を保証できないため自動マージを中止し、
         `merge_pull_request` を呼ばずに状況をユーザーへ報告して判断を仰ぐ。
   - [ ] **マージ後の成功判定はマージ方式に合わせる(実行した場合)。** `squash` / `rebase` では
         PR の head SHA が `main` の履歴に残らないため「head SHA が履歴に含まれる」では判定できない。
         `merge_pull_request` の応答(`merged: true` とマージコミット SHA)を確認し、続けて
         `pull_request_read`(`get`)で PR 状態が `MERGED`、かつ **マージ済みの head が事前
         スナップショットで控えた HEAD SHA と一致**することを確認する。想定と異なる head が
         マージされていた場合は続行せず、速やかにユーザーへ報告する。

## Phase 5. マージとブランチ削除

Phase 4 手順 8 のマージ直前チェックリスト(CI グリーン・
CodeRabbit 等の自動レビューボットの再レビュー完了・未解決スレッドゼロ)を
**すべて満たしてから**、以下を実施する:

1. **マージする。**`mcp__github__merge_pull_request` で PR を `main` にマージする。
2. **リモートブランチ削除は GitHub の自動削除に任せる(手動削除コマンドは実行しない)。**
   本リポジトリは「Automatically delete head branches」(`delete_branch_on_merge`)が
   **有効**なため、PR を GitHub 上(MCP のマージを含む)でマージすると、head ブランチは
   GitHub 側で自動的に削除される。よって **Claude 側でのリモートブランチ削除操作は不要**。

   > ⚠️ **Web 実行環境ではリモートブランチの手動削除はできない(必ず失敗する)。**
   > `git push origin --delete <branch>` はローカル git プロキシが `HTTP 403` で拒否し、
   > GitHub REST API の `DELETE /repos/.../git/refs/heads/<branch>` も egress プロキシが
   > 「Write access to this GitHub API path is not permitted through this proxy」で
   > `403` を返す。GitHub MCP にブランチ削除ツールも無い。
   > したがって削除は上記の自動削除に依存し、**手動削除コマンドは試みない**
   > (試みるとエラーになり、ワークフローが失敗扱いになるため)。

3. **ローカルの作業ブランチ**は不要なら削除してよい(`git branch -d <branch>`。ローカル操作は可能)。
4. **購読を解除する。**`unsubscribe_pr_activity` で PR の監視を終了する。
5. 完了をユーザーへ日本語で報告する(記事タイトル・PR 番号・マージ済みである旨)。
   自動削除の対象は head ブランチのみのため、マージ後もリモートに残るブランチがあれば
   その旨を伝える(下記参照)。

> **過去に溜まったブランチ / 自動削除の対象外:** 自動削除は「GitHub 上でマージされた PR の
> head ブランチ」だけが対象。PR を経ず push しただけのブランチ、close(未マージ)された PR の
> ブランチ、設定有効化より前にマージされたブランチはリモートに残る。これらは Web セッションからは
> 削除できないため、**GitHub の UI(PR 画面の「Delete branch」やブランチ一覧の削除)**、または
> ローカル環境から削除する。Claude からは削除できない旨をユーザーへ伝える。

## 全体フロー要約

```text
Phase 0  モード確認(オート/手動/PR 前まで)＋テーマ確認
Phase 1  記事作成(日英・ルール準拠)
Phase 2  サブエージェント反復レビュー(文体・構成 + 技術検証専任パス)  ── 指摘ゼロまでループ
Phase 3  モード別分岐:
           manual → 内容確認 → 承認で Phase 4 へ
           auto   → 確認スキップで Phase 4 へ
           pre_pr → コミット＆push まで実施して停止(PR 以降は指示待ち)
Phase 4  PR 作成 → CI 監視 ── グリーン＆指摘ゼロまで修正・再実行ループ
         (修正 push 後は CodeRabbit へ @coderabbitai review を要求、対応済みスレッドは解決済みにする、
          マージ前に未解決ゼロを確認。Copilot 再レビューは要求しない)
Phase 5  マージ → (head ブランチは GitHub が自動削除) → 購読解除 → 完了報告
```

## 判断の原則

- 指摘は件数を絞らない。見つけたものはすべて解消する。
- **対応を終えたレビュー指摘はスレッドを解決済みにする。** 修正で対応したもの、
  および反論・見送りで対応を終えたもの(理由を返信済み)は、いずれも
  `mcp__github__resolve_review_thread` で resolved にする。マージ前に未解決スレッドを残さない。
- **オートモードでは、マージを含めて確認を挟まず自動で完走する。** スキップするのは Phase 3 の PR 前確認だけでなく、
  マージ可否も含む。マージは Phase 4 手順 8 の基準(`main` が `protected: true` なら自動マージ、保護なしの場合のみ中止して確認)で判断する。
  オートモードで確認を残すのは、**修正方針に重大な曖昧さがある/破壊的で不可逆な内容判断が必要な場合に限る**(マージ操作そのものはこれに当たらない)。
- **途中経過はユーザーへ報告しない。** 技術検証の検索・裏取り(`Microsoft_Learn` 等)、セルフチェックインの予約/再予約、
  自動レビューボットの「処理中(review in progress / Currently processing)」待ちといった中間状態は、逐一報告しない。
  ユーザーへ表に出すのは、**選定理由(`article-auto` 由来の場合)・真の停止条件に該当した確認・最終の完了報告**に絞る。
- **`pre_pr`(PR 前まで)モードでは、コミット＆push で停止する。**
  PR 作成以降は明示的な指示があるまで一切行わず、勝手に PR を作成・マージしない。
- **リモートブランチの手動削除コマンドは実行しない**(Web 環境では 403 で必ず失敗する)。
  head ブランチの削除は GitHub の自動削除に任せる(Phase 5)。
- 手動モードでは、ユーザー承認を得るまで PR を作成しない。
